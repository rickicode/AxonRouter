# Tunnel system flattening and turbopackIgnore removal

The tunnel subsystem's 6-layer indirection chain (API route → access layer → facade → state manager runtime → status manager → connection runtime → cloudflared) is collapsed to 2-3 layers: API route → runtime module → cloudflared. Twelve dead wrapper files are deleted. The `new Function('m', 'return import(m)')` eval hack in four API routes is replaced with standard dynamic `import()`. Separately, all 72 inline `/*turbopackIgnore: true*/` comments are stripped from 20 source files, with a `turbopack.ignoreIssue` block added to `next.config.ts` to suppress NFT tracing warnings instead.

**Watch for:** The `ignoreIssue` config is a warning-suppression mechanism, not a fix for the underlying NFT trace fan-out — the user explicitly wants native Turbopack compatibility without any ignore/suppression mechanism (confirmed). The `loadRuntimeModule()` wrappers around bare `import()` in the API routes now serve no purpose (confirmed). The `tunnelStatus.ts` aggregator function receives `download` as `unknown` with no type narrowing (confirmed).

## High-level view

The dead-code removal is clean — zero remaining imports of deleted modules, `tsc --noEmit` passes. The consolidation into `state.ts` and `tunnelStatus.ts` is a net simplification: consumers import directly from the module that owns the logic rather than traversing re-export hops.

Replacing the `new Function` eval hack with plain `import()` is correct, but the `loadRuntimeModule` wrapper functions remain in four routes despite now being trivial one-liners. They no longer serve any bundler-evasion purpose and should be inlined.

The `turbopack.ignoreIssue` addition is a valid Next.js 15 config option, but it blanket-suppresses NFT diagnostics across broad directory trees (`src/lib/**`, `src/mitm/**`, `src/app/api/**`). Any future legitimate NFT problem in those trees will be silently swallowed. Per the user's stated goal of native Turbopack compatibility, the `ignoreIssue` config should not ship — the modules themselves should be made compatible rather than having their warnings suppressed.

## Dead-code removal and state consolidation

Twelve files removed, three functions consolidated into `state.ts`. The new exports (`loadTunnelStateSnapshot`, `resolveTunnelShortId`, `saveTunnelConnectionState`) preserve call-site semantics for the two runtimes (cloudflared, tailscale) that share the persisted state blob.

`tunnelStatus.ts` replaces a three-file chain (`tunnelStateStatusRuntime` → `tunnelStatusManager` → `tunnelConnectionStatusRuntime` + `tailscaleStatusRuntime`). One behavioral subtlety: the previous code awaited `getTunnelStateStatusRuntime()` — a lazy-cached promise that memoized the module reference — then called status functions. The new code does a fresh `import()` on every request. ES module imports are runtime-cached so this is functionally equivalent, but the explicit memoization layer is gone. If a future refactor makes these modules stateful at import time, the behavior would diverge.

## The `ignoreIssue` approach vs. native compatibility

The inline `/*turbopackIgnore: true*/` comments told Turbopack's static analyzer to skip tracing through specific call sites. Removing them means Turbopack now traces into those paths and emits NFT diagnostics. The `ignoreIssue` config suppresses *diagnostics* at the reporting layer — Turbopack still traces, it just doesn't warn.

The rules use broad globs: `**/src/lib/tunnel/**`, `**/src/mitm/**`, `**/src/app/api/**`. Any new module added under these trees that triggers a legitimate NFT diagnostic will be silently ignored. For a codebase targeting native Turbopack compatibility, this entire config block should be removed. The proper approach is restructuring modules so that:

- Top-level `path.join(os.homedir(), ...)` calls move into functions that are only invoked at runtime (already the case for most, but some like `DB_JSON_FILE` in `sqliteBootstrap.ts` are top-level constants).
- Dynamic `import()` in API routes is recognized by Turbopack as intentional code-splitting (which it already is — the issue is likely the transitive trace into `cloudflared.ts` which spawns child processes).

The `ignoreIssue` block should not ship if the goal is zero suppression.

## API route `loadRuntimeModule` wrappers

All four tunnel API routes keep `function loadRuntimeModule() { return import("@/lib/tunnel/...") }`. Before this change, the function used `new Function('m', 'return import(m)')` to break static analysis — that was its reason to exist. With plain `import()`, the wrapper adds nothing: it doesn't memoize, doesn't abstract, and doesn't isolate errors (the caller already has try/catch). Inline the import into the handler body.
