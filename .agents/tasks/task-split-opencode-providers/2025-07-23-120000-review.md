# Splitting OpenCode into tiered providers (free / Go / Zen)

The single `opencode` provider definition has been decomposed into three tiers with distinct access levels. The free tier (`opencode`) loses its `passthroughModels` and `modelsFetcher` config, locking it to 4 hardcoded models. The mid-tier (`opencode-go`) gets an expanded 16-model list matching OmniRoute's subscription offering. A new top tier (`opencode-zen`) is added to `APIKEY_PROVIDERS` with `passthroughModels: true` and `modelsFetcher` pointing at the same `/zen/v1/models` endpoint the free tier previously used, giving it full 43-model dynamic discovery.

**Watch for:** The `modelsFetcher` on `opencode-zen` uses type `"opencode-free"` — this looks like a leftover from when it belonged to the free provider and may confuse the filtering logic in the suggested-models API route (confirmed). The `opencode-go` provider's Qwen models gained `targetFormat: "claude"` which changes request formatting for existing subscribers on deploy (confirmed).

## High-level view

The free tier is now a simple no-auth provider with a static model list. Removing `modelsFetcher` means `noAuthSync.ts` will no longer auto-fetch models for it — free users see exactly 4 models, and adding new ones requires a code deploy.

The Go tier's model list was reordered and expanded from ~8 to 16 entries, adding MiMo V2.5 variants, Qwen 3.7 Max, Hunyuan3, and DeepSeek V4 Pro/Flash. Qwen 3.6 Plus and Qwen 3.5 Plus gained `targetFormat: "claude"` annotations that weren't there before — these will change how requests to those models are formatted at the executor layer.

OpenCode Zen is the new premium tier carrying over the dynamic model discovery that the free tier previously had. Its `modelsFetcher` retains `type: "opencode-free"` from the original config. Whether the suggested-models API route serves the correct (full) model set for a paid tier depends on how it interprets this type string.

## modelsFetcher type inheritance on Zen

The `opencode-zen` provider definition carries `modelsFetcher: { url: "https://opencode.ai/zen/v1/models", type: "opencode-free" }`. The `type` field flows into `providerModelsFetcher.ts` which passes it as a query param to `/api/providers/suggested-models?type=opencode-free`. If that API route uses the type to scope results (e.g., filtering to only free-tier models), Zen's dynamic model list would be artificially limited despite having `passthroughModels: true`.

The hardcoded 43-model list in `PROVIDER_MODELS["opencode-zen"]` acts as a fallback/initial catalog, but the `modelsFetcher` is meant to provide live updates. If the type string causes filtering, the live updates would be a subset of what the hardcoded list promises.

```typescript
// providers.ts — the type string is "opencode-free" on a paid provider
"opencode-zen": {
  ...
  modelsFetcher: { url: "https://opencode.ai/zen/v1/models", type: "opencode-free" },
}
```

Consider renaming to `type: "opencode-zen"` or `type: "opencode-all"` to make the intent explicit, unless the API route ignores type for filtering purposes.

## targetFormat changes on Go-tier models

The diff adds `targetFormat: "claude"` to `qwen3.6-plus` and `qwen3.5-plus` in the Go tier. These models previously had no targetFormat annotation — they would have been formatted using the default (OpenAI-compatible) format. After this change, the executor layer will send Claude-format messages to these models.

This is a behavioral change for existing opencode-go users on deploy. If the OpenCode backend already expects Claude format for these models (likely, since OmniRoute's registry has the same annotation), this is correct. If not, Qwen requests from Go subscribers will break silently — the backend will receive an unexpected message format.
