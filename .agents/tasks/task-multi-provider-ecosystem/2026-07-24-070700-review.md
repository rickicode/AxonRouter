# Multi-provider ecosystem: categorization, OpenCode variants, and test workflows

This change expands the provider dashboard with a category filter bar, per-card test buttons, and a global "Test All" action. It also adds two new OpenCode executor variants (Zen and Provider), wires them into the routing layer, and fixes the opencode-go `CLAUDE_FORMAT_MODELS` set to include qwen3.6-plus and qwen3.5-plus (matching their `targetFormat: "claude"` annotations). A cross-layer consistency test validates alias uniqueness, executor registration, and routing config presence.

Watch for: singleton executor `_lastModel` race condition under concurrent requests (confirmed); filter bar "Compatible" chip is wired but has no effect on the dynamically-loaded compatible provider cards (confirmed).

## High-level view

The four OpenCode variants (Free/oc, Go/ocg, Zen/ocz, Provider/ocp) are now fully wired end-to-end: each has its own provider definition, model list, routing config, and dedicated executor. The executors share an identical pattern — a `CLAUDE_FORMAT_MODELS` set that selects between `/messages` and `/chat/completions` endpoints, with header switching between `x-api-key` (Anthropic style) and `Authorization: Bearer` (OpenAI style). The opencode-go fix adds qwen3.6-plus and qwen3.5-plus to its claude-format set, correcting a routing mismatch where these models had `targetFormat: "claude"` in providerModels but were being sent to the OpenAI endpoint.

The UI layer adds a `ProviderFilterBar` with category chips and a `ProviderCategoryBadge` on each card. API Key providers are deterministically sorted with `apiKeyCompatible: true` entries first, alphabetically within each group. The "Compatible" filter chip is present but doesn't control visibility of the dynamically-loaded OpenAI/Anthropic-compatible provider section.

The test infrastructure adds per-card Zap buttons and a "Test All Providers" header button. `ProviderTestContract` declares four phases with `getProviderTestCapabilities()` determining applicability by auth type. Tests cover variant identity, routing config presence, category logic, ecosystem consistency, and capability contracts — but not the actual HTTP routing behavior or the `_lastModel` race under concurrency.

<summary>Issues (3)</summary>

1. **Singleton executor `_lastModel` race** — `OpenCodeZenExecutor`, `OpenCodeProviderExecutor`, and `OpenCodeGoExecutor` store the current model on `this._lastModel` between `buildUrl()` and `buildHeaders()`. Because executors are singletons, concurrent requests will clobber each other's model state. Pass the model through the call chain or use a request-scoped context instead of instance state.
2. **Filter bar "Compatible" chip dead path** — The "compatible" filter checks `getProviderCategory(providerId)` for "OpenAI Compatible" or "Anthropic Compatible", but these providers are dynamically loaded from `providerNodes` and rendered in a separate grid that doesn't go through `categoryFilterFn`. The filter chip has no effect on those cards.
3. **`apiKeyCompatible` field without runtime impact** — The field is purely cosmetic (controls sort order). The 30 providers flagged were chosen without an obvious distinguishing criterion (e.g., `mimo` is false but uses standard Bearer auth). Document the field's contract or rename to `uiSortPriority`.

## Singleton executor race on `_lastModel`

All three new/modified executors use this pattern:

```ts
buildUrl(model) {
  this._lastModel = model;
  return CLAUDE_FORMAT_MODELS.has(model) ? `${BASE}/messages` : `${BASE}/chat/completions`;
}

buildHeaders(credentials, stream = true) {
  if (CLAUDE_FORMAT_MODELS.has(this._lastModel!)) {
    headers["x-api-key"] = key;
    headers["anthropic-version"] = "2023-06-01";
  } else {
    headers["Authorization"] = `Bearer ${key}`;
  }
}
```

The `executors` map creates one instance per provider:

```ts
const executors = {
  "opencode-zen": new OpenCodeZenExecutor(),
  "opencode-provider": new OpenCodeProviderExecutor(),
};
```

If request A calls `buildUrl("minimax-m2.5")` (claude-format), then request B calls `buildUrl("glm-5")` (openai-format) before A reaches `buildHeaders`, request A will send `Authorization: Bearer` to the `/messages` endpoint — a fail-open mismatch that returns an upstream 4xx.

**Confidence: confirmed.** The singleton pattern is in `open-sse/executors/index.ts` lines 34-36; the `_lastModel` write/read is in each executor file.

This is the same pre-existing pattern from `OpenCodeGoExecutor`, but the PR extends it to two additional executors, widening the race surface.

## Filter bar "Compatible" chip dead path

The `categoryFilterFn` in `page.tsx` checks `getProviderCategory(providerId)` for `"OpenAI Compatible"` or `"Anthropic Compatible"`. But those providers come from `providerNodes` (fetched from `/api/provider-nodes`) and render in a separate grid section that never calls `categoryFilterFn`. Selecting "Compatible" hides all static sections (none match) while the compatible section stays visible regardless.

**Confidence: confirmed.** The compatible providers grid is rendered unconditionally and doesn't reference `categoryFilterFn` or `activeFilter`.
