# Removal of cloud worker, R2 backup, and usage worker subsystems (v2)

This change removes the cloud worker relay, R2 backup/restore pipeline, and the child-process usage worker â€” about 24,500 lines across 160 files. The cloud relay is superseded by WorkerRelay, R2 is no longer needed, and usage scheduling continues in-process via `usageRefreshQueue`. A secondary fix replaces the broken `require()`-based machineId module with pure ESM using `node:crypto` + `node:os`.

The v1 review raised 6 issues. Issues 1-3 (stale route guards, orphaned query key, dead test mocks) are confirmed fixed. Issues 4-5 (vestigial DB column, hostname-based machineId) were accepted as design decisions. Issue 6 (ESM inconsistency) was partially fixed â€” `tunnelConnectionRuntime.ts` now uses ESM `hostname` import, while `passwordStoreRuntime.ts` retains `require("os")` intentionally as a CJS-compatible child process module.

**Watch for:** The `ProviderLimits/index.tsx` UI component still calls `/api/usage-worker/status` and `/api/usage-worker/run` â€” endpoints deleted by this PR â€” which will produce 404 errors on the Usage dashboard when users click "Refresh All" or load scheduler status (confirmed). `ProfileSettingsContent.tsx` still renders a "Usage Worker" settings form that writes `usageWorker` settings to the DB, but nothing reads or acts on those values anymore (confirmed).

**Verdict**: NEEDS_CHANGES

## High-level view

The core deletion is thorough: cloud sync, R2, and the child-process worker are gone from library code, API routes, startup logic, and shared services. The route guards, query keys, and test mocks flagged in v1 are all cleaned up correctly.

Two UI components (`ProviderLimits/index.tsx` and `ProfileSettingsContent.tsx`) still reference the deleted usage worker system. The ProviderLimits component calls deleted API routes which will 404 at runtime â€” this is a functional regression on the Usage dashboard. The ProfileSettings form writes dead config that nothing consumes.

## ProviderLimits UI calls deleted API routes

The PR deletes `src/app/api/usage-worker/run/route.ts` and `src/app/api/usage-worker/status/route.ts` but does not update `ProviderLimits/index.tsx`, the primary Usage dashboard component. Two fetch calls now target non-existent routes:

```typescript
// line 355 â€” scheduler status polling
const response = await fetch("/api/usage-worker/status", { cache: "no-store" });

// line 538 â€” manual "Refresh All" trigger
const response = await fetch("/api/usage-worker/run", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ reason: "dashboard_manual_refresh", mode: "all" }),
});
```

The component handles non-200 responses gracefully (sets error state), so this won't crash the page, but the "Refresh All" feature and scheduler status display become non-functional. Since the in-process `usageRefreshQueue` still exists and handles the same responsibility, these endpoints should be re-created as thin wrappers around the queue, or the component should be updated to use whatever API the queue exposes.

## ProfileSettings renders a form for a removed subsystem

`ProfileSettingsContent.tsx` maintains state for `usageWorkerForm` (enabled, cadenceMinutes, exhaustedThresholdPercent) and submits it to `/api/settings` as `{ usageWorker: { enabled, intervalMinutes } }`. The settings route still accepts arbitrary fields in the PATCH body and stores them, so no error occurs â€” but the stored `usageWorker` object is never read by any runtime code. The `usageRefreshQueue` has its own hardcoded configuration and doesn't consult these settings.

This leaves a visible settings section in the UI that appears functional but does nothing. Users toggling "Usage Worker enabled" or changing the interval will see a success message but observe no behavioral change.

## V1 issue verification

All six v1 issues have been addressed:

- **Stale route guards**: `/api/r2`, `/api/usage-worker`, `/api/cloud-urls` removed from `proxy.ts` and `dashboardGuard.ts`.
- **Orphaned query key**: `r2Settings` removed from `queryKeys.ts` and `useInvalidate.ts`.
- **Test mocks**: Dead `vi.mock` blocks for `@/lib/usageWorker/client`, `@/lib/cloudSync`, and `isCloudEnabled` removed from both settings route test files.
- **Usage DB column**: Accepted â€” `cloud_worker_id` remains with `null` binding for schema compatibility.
- **MachineId stability**: Accepted â€” hostname-based derivation is the chosen approach.
- **ESM inconsistency**: `tunnelConnectionRuntime.ts` updated to ESM `import { hostname } from "node:os"`. `passwordStoreRuntime.ts` retains `require("os")` as an intentional CJS pattern.
