# Removal of cloud worker, R2 backup, and usage worker subsystems

This change guts three major subsystems — the cloud worker sync infrastructure, the R2 backup/restore pipeline, and the isolated usage worker process — reducing the codebase by ~24,500 lines across 153 files. The motivation is simplification: the cloud worker relay is superseded by WorkerRelay, R2 backups are no longer needed, and usage scheduling is more efficient in-process (via `usageRefreshQueue`) for binary production deployments. A secondary fix replaces the broken `require()`-based machineId module with a pure ESM implementation using `node:crypto` + `node:os`.

**Watch for:** Stale route references in `proxy.ts` and `dashboardGuard.ts` will cause 404s to pass through auth guards unnecessarily (confirmed). Test files still mock removed modules (`@/lib/usageWorker/client`, `@/lib/cloudSync`, `isCloudEnabled`) which will break or produce false-pass results on import resolution changes (confirmed). The `cloud_worker_id` column is still referenced in the usage DB INSERT statement even though no code ever populates it with a real value anymore (confirmed). Machine ID stability regressed from hardware-based to hostname-based, which changes on container restarts (likely).

## High-level view

The deletion scope is clean and comprehensive for the library and route layers. Cloud sync, cloud URL management, cloud usage polling, R2 backup/restore/publish, and the child-process usage worker are all gone from production code paths. The `initializeApp` startup sequence is noticeably simpler — it no longer conditionally spawns background processes or probes cloud state.

The machineId replacement trades a native binary dependency for a zero-dep hostname hash. This fixes the ESM compatibility issue but changes the stability contract: hostnames are mutable and non-unique in containerized environments, which affects tunnel registration identity and MITM password encryption key derivation.

Three files in the routing/guard layer (`proxy.ts`, `dashboardGuard.ts`, `queryKeys.ts`) still reference deleted API paths (`/api/r2`, `/api/usage-worker`, `/api/cloud-urls`). These are dead entries that won't crash the app but will mislead future editors and waste auth middleware cycles on non-existent routes.

Two test files (`caveman-settings-route.test.ts`, `morph-settings-route.test.ts`) still mock `@/lib/usageWorker/client` and `@/lib/cloudSync` — modules that no longer exist. These tests pass today only because Vitest's `vi.mock` intercepts the import before resolution fails, but they're fragile and will break on any Vitest version that changes mock resolution order.

## Incomplete reference cleanup in route guards

`src/proxy.ts` (line 16) and `src/dashboardGuard.ts` (line 30) still include `/api/r2`, `/api/usage-worker`, and `/api/cloud-urls` in their protected-route arrays. Since the route handlers have been deleted, requests to these paths will 404 after passing through auth middleware. The entries should be removed to keep the guard lists truthful.

Similarly, `src/shared/query/queryKeys.ts` still exports `r2Settings: () => ["r2"] as const`. No UI code uses this anymore.

## MachineId: hostname as identity anchor

The old implementation used `node-machine-id` which reads `/etc/machine-id` or the Windows registry — a value that persists across reboots and process restarts. The replacement hashes `os.hostname()`, which is:

- Mutable via `hostnamectl` or container orchestration
- Non-unique across replicas in Docker Compose / Kubernetes
- Changes when a container is recreated without an explicit hostname

Two runtime consumers depend on stable machine IDs: tunnel registration (`tunnelConnectionRuntime.ts` line 33) uses it to derive a short ID for the relay worker, and MITM password encryption (`passwordStoreRuntime.ts` line 23) uses it to derive the AES key. If the hostname changes between restarts, previously encrypted MITM passwords become unrecoverable — the decryption silently returns `null` and the password is lost.

The fallback path in `getConsistentMachineId` is also worth noting: if `hostname()` throws (extremely unlikely on any supported platform), it falls back to `Date.now() + Math.random()`, producing a different ID on every call. This is a non-recoverable state for anything that persists the derived value.

```typescript
// tunnelConnectionRuntime.ts — now uses hostname
function getMachineId() {
  try {
    const raw = require("os").hostname();
    return crypto.createHash("sha256").update(raw + MACHINE_ID_SALT).digest("hex").substring(0, 16);
  } catch {
    return crypto.randomUUID().replace(/-/g, "").substring(0, 16);
  }
}
```

A more stable approach would read `/etc/machine-id` on Linux (available in all non-minimal containers) and fall back to a persisted random seed in the app's data directory.

## Usage DB: vestigial cloud_worker_id column

The `usage_events` table schema still includes `cloud_worker_id` as a column, and `writer.ts` still binds a positional `null` to it. The event normalization in `events.ts` correctly dropped `cloudWorkerId` from the normalized object, but the INSERT statement needs the positional `null` to match the column count. This works but leaves a permanently-NULL column in the schema that will confuse future readers and wastes a small amount of storage per row.

## Test files mocking deleted modules

Two test files still function only because `vi.mock` intercepts module resolution before the runtime attempts to load the real module:

```typescript
// caveman-settings-route.test.ts
vi.mock("@/lib/usageWorker/client", () => ({
  getUsageWorkerClient: () => ({ getStatus: getUsageWorkerStatus }),
}));
vi.mock("@/lib/cloudSync", () => ({ syncToCloud }));
```

The settings route itself no longer imports these modules, so the mocks are dead code that could mask import errors in adjacent modules. The `isCloudEnabled` mock in the `@/lib/localDb` mock block is also stale — `localDb.ts` no longer exports that function.
