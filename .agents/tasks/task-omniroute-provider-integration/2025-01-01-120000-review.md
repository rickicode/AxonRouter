# OmniRoute provider catalog integration and dashboard categorization

AxonRouter's provider registry gains 100+ new providers from OmniRoute, organized into six new category exports (LOCAL, SEARCH, AUDIO_ONLY, UPSTREAM_PROXY, CLOUD_AGENT, SYSTEM) alongside the existing FREE/FREE_TIER/OAUTH/APIKEY/WEB_COOKIE taxonomy. The dashboard page is split from a single 1975-line file into four modules and renders providers in dedicated visual sections with "Free" badges. Backward compatibility for `FREE_TIER_PROVIDERS` and all pre-existing provider IDs is preserved.

**Watch for:** Duplicate `brave-search` key across APIKEY_PROVIDERS and SEARCH_PROVIDERS silently changes its alias when AI_PROVIDERS is built (confirmed). Several textIcon values collide across providers ("CF" used by comfyui, cloudflare-ai, and clarifai), creating ambiguous fallback icons on the dashboard (confirmed). `isLocalProvider()` and `isSelfHostedChatProvider()` are functionally identical but named differently, which will confuse future contributors (confirmed).

## High-level view

The provider registry is a single 628-line file organized into clear section-separated category maps. New helper functions (`isLocalProvider`, `providerAllowsOptionalApiKey`, `supportsBulkApiKey`) gate routing behavior, and sub-category Sets (`IMAGE_ONLY_PROVIDER_IDS`, `AGGREGATOR_PROVIDER_IDS`, etc.) power dashboard filtering. Each category is its own exported object, composed into `AI_PROVIDERS` at the bottom. The key concern is a duplicate key (`brave-search`) that causes silent alias overwriting.

The dashboard refactor splits modal dialogs and card components into separate files, keeping the page under 1100 lines. The `filteredApiKeyProviders` list filters by `serviceKinds.includes("llm")`, which means non-LLM providers in APIKEY_PROVIDERS (like `brave-search` with `serviceKinds: ["webSearch"]`) don't render in the API Key section — they only appear in their dedicated category section. This avoids double-rendering but doesn't fix the data-layer alias overwrite.

## Duplicate key: `brave-search` in two categories

The `brave-search` provider key is defined in both `APIKEY_PROVIDERS` (line 207, alias "brave") and `SEARCH_PROVIDERS` (line 375, alias "brave-search"). JavaScript object spread resolves last-writer-wins:

```typescript
export const AI_PROVIDERS = {
  ...APIKEY_PROVIDERS,     // brave-search: { alias: "brave", ... }
  ...SEARCH_PROVIDERS,     // brave-search: { alias: "brave-search", ... }  <-- wins
};
```

Any existing user routing to `brave` via `getProviderByAlias("brave")` will now fail — the `ALIAS_TO_ID` map will not contain the "brave" key. The SEARCH version also adds `hasFree: true` and `authHint` that the APIKEY version lacks, so existing code relying on the APIKEY definition's shape may behave differently.

On the dashboard, `brave-search` in APIKEY_PROVIDERS has `serviceKinds: ["webSearch"]` so it's filtered out of the API Key section (which requires `"llm"`). It renders only in the Search Providers section via `filteredSearchProviders`. The duplicate doesn't cause double-rendering, but the data-layer alias change is still a breaking concern for routing.

## textIcon and alias collisions

Three providers share textIcon "CF": comfyui, cloudflare-ai, clarifai. Two share "SP": serper, serper-search. Two share "DB": databricks, doubao. These are used as fallback text when provider PNG icons fail to load. With 100+ providers, icon load failures become more likely — collisions mean a user can't distinguish which provider they're looking at from the fallback alone.

The `alias` field has no collisions (verified), so routing is not affected. This is a UX concern, not a correctness one.

## Functional overlap between `isLocalProvider` and `isSelfHostedChatProvider`

`LOCAL_PROVIDERS` contains exactly the same nine keys as `SELF_HOSTED_CHAT_PROVIDER_IDS`: lm-studio, vllm, lemonade, llamafile, llama-cpp, triton, docker-model-runner, xinference, oobabooga. The `providerAllowsOptionalApiKey` function calls both functions independently via `||`, which is logically redundant today. If these concepts are intended to diverge (e.g., adding a local image-only provider that isn't "self-hosted chat"), the naming distinction makes sense — but that intention isn't documented anywhere.

## Untyped helper functions

All new helper functions use bare `providerId` parameters without TypeScript annotations:

```typescript
export function isLocalProvider(providerId) { ... }
export function supportsBulkApiKey(providerId) { ... }
```

The file already exports TypeScript types (`AiProviderId`, `ServiceKind`). Adding parameter types would catch call-site mistakes and is consistent with the typed approach used elsewhere in the file.

## Implicit LLM filter hides non-LLM APIKEY providers

The `filteredApiKeyProviders` filter (`serviceKinds.includes("llm")`) implicitly hides non-LLM API key providers (audio, search, image-only) from the main API Key section. Those providers appear in their dedicated sections instead. There's no inline comment explaining this behavior — a future contributor adding a new `webSearch` provider to APIKEY_PROVIDERS would wonder why it doesn't render in the API Key section.
