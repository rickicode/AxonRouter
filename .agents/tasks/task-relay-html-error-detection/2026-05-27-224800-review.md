# Relay and HTTP proxy HTML error detection with account overrides display fix

This change intercepts HTML error responses from relay proxies and HTTP proxies (Cloudflare blocks, nginx error pages, rate-limit splash pages) and treats them as proxy failures rather than forwarding them as if they were legitimate provider responses. The detection checks both status code (403/429/5xx) and content-type (`text/html`), then applies the existing `strictProxy` fallback logic: throw if strict, fall back to direct if not. A parallel fix in the proxy pools dashboard limits the Bindings tab to accounts with explicit proxy overrides by default.

**Watch for:** The re-throw guard (`if (error.phase)`) can accidentally swallow non-diagnostic errors that happen to have a `.phase` property (confirmed). The `isRelayHtmlError`/`isProxyHtmlError` functions silently pass through error responses that lack a `content-type` header entirely, which some relay software omits (likely).

## High-level view

The HTML error detection is a response-level interception inserted between receiving the proxy response and returning it to the caller. Two near-identical helper functions (`isRelayHtmlError`, `isProxyHtmlError`) gate on status + content-type, and the surrounding try/catch blocks gain a re-throw sentinel to distinguish diagnostic errors from network failures. The detection is conservative: it requires explicit `text/html` in the content-type header and an error status code, so legitimate provider 4xx/5xx with JSON bodies pass through untouched.

The account overrides fix adds a filtering layer and a UI toggle. By default the Bindings tab now shows only connections that have a `proxyPoolId` or `proxyGroupId` in their `providerSpecificData`, with a "Show all accounts" switch to reveal the full list when assigning new overrides.

## Missing content-type gap in detection logic

When `response.headers.get("content-type")` returns `""` (header absent), neither the JSON nor the SSE nor the HTML check matches, so the function returns `false`. A relay returning `<html>Rate Limited</html>` with no content-type header would pass through to the caller as if it were a valid provider response. Many lightweight reverse proxies and CDN edge nodes serve error pages without explicit content-type headers, making this the most likely false-negative scenario in production.

## Re-throw sentinel for diagnostic errors

After the HTML detection throws inside the try block, the outer catch needs to distinguish this from a network-level failure:

```typescript
if (relayError instanceof Error && (relayError as any).phase) {
  throw relayError;
}
```

This is a duck-type check on a common property name. Any error object with a `.phase` string property would be re-thrown without the `strictProxy` logic or the fallback-to-direct path applying. A tighter guard (check for specific phase prefixes like `relay-html-error` or `proxy-html-error`, or use a private symbol) would eliminate the ambiguity.

## Test gaps

Not tested: the case where `content-type` is absent (empty string), the case where the direct fallback itself fails after HTML error detection, and the unconsumed response body scenario. The MITM-bypass code path also uses the dispatcher but doesn't have HTML detection, which appears intentional since MITM bypass is a different trust domain.
