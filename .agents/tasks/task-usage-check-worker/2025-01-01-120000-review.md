# Periodic usage-check background scheduler

A new `UsageCheckScheduler` refreshes usage/quota for all active OAuth provider connections on a 5-minute timer. It follows the same singleton-on-global, setTimeout-with-unref, bootstrap-gate pattern as the existing `ModelSyncScheduler`. A status API exposes the scheduler state, and the Quota dashboard renders an inline worker status bar with manual-trigger capability.

**Watch for:** The scheduler is always-on with no `loadSettings()` / DB-backed disable path (confirmed), so once bootstrapped there is no way to stop it at runtime short of a redeploy. The `POST /api/usage-worker/status` endpoint calls `runScheduled()` which respects the `this.running` guard but returns `lastRun` (possibly null) silently if the scheduler is already active — the caller gets a 200 with `null` body on the first-ever manual trigger while a scheduled run is in-flight (confirmed). The bootstrap module-level `let bootstrapped` flag resets on Next.js hot-reload in dev, allowing duplicate timers against the global singleton (likely — same risk exists in the ModelSync counterpart, so consistent but worth noting).

## High-level view

The scheduler is a clean port of the ModelSync pattern: global singleton, `setTimeout.unref()`, `Promise.allSettled` over connections, dedup via the existing `runDedupedUsageRefreshJob`. The key simplification relative to ModelSync is the absence of a `loadSettings()` call — settings are hardcoded at construction time, so the `enabled` field reported in the status response is always `true` and cannot be toggled by an admin.

The API route is minimal: GET returns scheduler status behind `requireManagementAuth`, POST triggers a manual run. The POST handler awaits `runScheduled()` and returns its result, which means the HTTP response is held open for the full duration of all connection refreshes — potentially dozens of seconds if many connections are active.

The UI component polls `/api/usage-worker/status` every 30 seconds and ticks a re-render every 5 seconds to keep the relative-time labels fresh. It degrades gracefully to invisible when the endpoint isn't reachable or unauthenticated.

<summary>Issues (5)</summary>

1. **No runtime disable path** — `settings.enabled` is hardcoded `true` with no `loadSettings()` from DB. Add a settings lookup (or remove the field from the status response to avoid implying configurability).
2. **POST response may be null** — If a scheduled run is already active, `runScheduled()` returns `this.lastRun` which is `null` before the first completed run. The API returns `200 null` with no indication that the request was a no-op. Return a distinct response (e.g., `{ skipped: true, reason: "already running" }`).
3. **POST holds connection open** — `runScheduled()` awaits all connection refreshes. With many active connections this can exceed typical HTTP timeout budgets. Consider returning 202 Accepted and letting the caller poll status.
4. **Unbounded parallelism** — All OAuth connections are refreshed concurrently via `Promise.allSettled`. For installations with dozens of connections this may spike outbound request volume. A concurrency cap (e.g., `p-limit`) would be safer.
5. **Bootstrap flag vs hot-reload** — The module-level `let bootstrapped` resets on HMR, but the global singleton persists, so `start()` is called again on an already-running scheduler. `start()` calls `scheduleNext()` which clears the old timer first, so this is safe but creates a spurious reschedule.

## Missing `loadSettings()` and the always-enabled scheduler

The `ModelSyncScheduler` loads settings from the database on every `start()` and again at the top of `runScheduled()`, allowing an admin to disable sync at runtime. `UsageCheckScheduler` hardcodes `{ enabled: true, intervalMinutes: 5 }` in the constructor and never reads from the DB. The `enabled` field is still exposed in `getStatus()`, giving the API consumer the impression that it can be toggled — the asymmetry with the reference pattern should be intentional or the field removed.

## POST semantics and the `runScheduled()` guard

`runScheduled()` begins with:

```typescript
if (this.running) {
  this.logger.warn?.("[UsageCheck] Run already active, skipping scheduled tick");
  return this.lastRun;
}
```

When the POST handler calls `runScheduled()` while a background tick is already active, the caller receives `this.lastRun`. Before the scheduler has ever completed a run, `lastRun` is `null`, so the API returns `200 null`. There's no way for the client to distinguish "your request was processed" from "your request was silently dropped." The ModelSyncScheduler has the same guard, but it's only called from internal scheduled ticks — exposing it via HTTP requires clearer response semantics.

## Parallel fan-out without concurrency control

The scheduler issues `Promise.allSettled` across all matching connections simultaneously. `runDedupedUsageRefreshJob` deduplicates per-connection but doesn't limit how many connections can be in-flight concurrently. An installation with 30+ OAuth connections will fire 30+ concurrent outbound HTTP requests to provider APIs in a single tick.

## Unauthenticated bootstrap trigger

The `ensureUsageCheckSchedulerStarted().catch(() => {})` call in `GET /api/usage/[connectionId]` has no auth gate — any request to this route triggers scheduler bootstrap. The scheduler itself doesn't expose data to the triggering request, but an external caller can force the scheduler into existence (and its periodic outbound requests) without credentials.
