# Round-robin proxy groups with health checks and dashboard UI

This change introduces proxy groups as a layer above individual proxy pools, allowing multiple proxies to be combined with round-robin or sticky selection modes. The resolution chain is extended to a five-level priority system: connection group > connection pool > provider-default group > provider-default pool > none. A background health check scheduler runs every 30 minutes, marks failing pools as "error", and the group resolver automatically skips them. The dashboard gets a tabbed UI with group CRUD, enhanced provider bindings with optgroups, and response-time badges.

**Watch for:** Deleted proxy pools leave stale IDs in group `proxyPoolIds` arrays (confirmed). In-memory round-robin cursors and sticky state are lost on process restart and are not concurrency-safe under concurrent requests (confirmed). The `isRunning` guard in the health check can silently drop a manual trigger if one is already in progress (confirmed).

## High-level view

The group data model stores an ordered list of pool IDs (`proxyPoolIds`). When a pool is deleted via the existing pools API, no cascade removes it from groups. The resolver tolerates this at runtime (null pool lookups are filtered out), but the stale reference persists in the database and will appear in the group edit UI until manually removed.

In-memory state (round-robin cursors and sticky tracking) lives in module-level Maps. State resets on every server restart or hot-reload, and concurrent requests can read the same cursor before either increments it, causing duplicate pool selections.

The health check's `isRunning` mutex prevents overlapping runs but means a manual trigger during an automated run returns immediately with empty results and no user-visible indication that it was a no-op.

When `resolveProxyFromGroup` returns null (strict group, all pools offline), the resolution chain falls through to the next priority level rather than failing the request. This dilutes "strict" semantics: a strict group with all-offline pools is silently bypassed if a lower-priority proxy exists.

## Strict group fallthrough ambiguity

When `resolveProxyFromGroup` returns null because all pools have `testStatus === "error"` and `strictProxy === true`, the caller (`resolveConnectionProxyConfig`) interprets null as "this level couldn't resolve" and falls through to the next priority level. A strict group that can't serve is silently bypassed rather than failing the request:

```typescript
// connectionProxy.ts: if all pools error + strictProxy
if (group.strictProxy === true) {
  return null;  // falls through to next priority level
}
```

If the intent of strictProxy is "never fall back to direct", the current behavior satisfies that (it won't go to direct, it'll try a lower-priority pool). But if the intent is "fail if this specific group can't serve", the chain should propagate an error rather than returning null.

## Stale pool IDs after deletion

When a pool is deleted via `DELETE /api/proxy-pools/[id]`, `cascadeRemoveProxyPoolReferences` clears `proxyPoolId` from connections and `providerProxyDefaults`, but does not touch any group's `proxyPoolIds` array. The stale ID remains.

At resolution time, the filtering happens before selection (`activePools` only includes pools that loaded successfully), so there's no functional impact on proxy routing. The issue is data cleanliness: the group edit UI will show a phantom entry, and the stale ID persists indefinitely until manual removal.

Fix: in the pool DELETE handler, after `cascadeRemoveProxyPoolReferences`, iterate all groups and filter out the deleted pool ID:

```typescript
const groups = await getCurrentProxyGroups();
for (const group of groups) {
  if (group.proxyPoolIds?.includes(id)) {
    await updateCurrentProxyGroup(group.id, {
      proxyPoolIds: group.proxyPoolIds.filter(pid => pid !== id)
    });
  }
}
```

## In-memory state and concurrency

```typescript
const roundRobinCursors = new Map<string, number>();
const stickyState = new Map<string, { poolId: string; count: number }>();
```

The cursor read and write are separated by multiple `await` calls (pool lookups):

```
read cursor → await pool lookups → write cursor+1
```

If two requests enter simultaneously, both read cursor=0, both select pool[0], both write cursor=1. Net effect: two consecutive requests hit the same pool, then one pool is skipped. For round-robin this is "mostly round-robin" — acceptable. For sticky mode, the count could be incremented by both requests before either writes back, halving the effective sticky duration under load.

Process restart clears all state. For sticky mode this means a user mid-sticky-window gets reset immediately.

## Health check scheduler edge cases

```typescript
if (isRunning) {
  return { checkedAt: lastHealthCheckAt || new Date().toISOString(), results: [] };
}
```

The POST endpoint returns `{ ok: true, checkedAt, results: [] }` — the dashboard `handleRunHealthCheck` calls `notify.success("Health check completed")` regardless of whether results are empty. The user gets a success notification for a no-op.

The sequential loop means a single slow pool (8-10 second timeout) blocks all subsequent checks. With many pools, a single run could take minutes.
