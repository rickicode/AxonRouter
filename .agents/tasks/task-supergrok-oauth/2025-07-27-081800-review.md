# SuperGrok/xAI OAuth provider mode with PKCE flow and image generation

This change adds a SuperGrok OAuth pathway so that xAI (Grok) users with a SuperGrok subscription can authenticate via OAuth PKCE instead of an API key. The implementation spans OAuth configuration, a headless import route, token refresh wiring in the executor layer, xAI image generation with size-to-aspect_ratio mapping, provider model/UI updates, and 23 unit tests across 4 files. The approach reuses the existing `xai` provider identity and adds a `supergrok_oauth` auth type so routing and credential resolution happen through the same paths.

**Watch for:** hardcoded client ID in `testUtils.ts` that drifts from the central config (confirmed); `n` parameter silently dropped for xAI image generation meaning multi-image requests produce a single image without error (confirmed); import route stores connection even when `refreshToken` is absent, creating permanently dead connections after expiry (likely); error text from upstream token server may leak to callers (confirmed).

## High-level view

The OAuth plumbing follows the project's existing PKCE pattern (Codex, Claude, Qwen): config constants, a provider entry with `buildAuthUrl`/`exchangeToken`/`mapTokens`, and a refresh path via `refreshWithForm`. The import route is the primary integration surface for headless clientsâ€”it refreshes, validates against `/v1/models`, and stores the connection.

Image generation for xAI maps OpenAI-style `size` values to xAI's `aspect_ratio` parameter. The xAI image API returns OpenAI-compatible JSON, so no dedicated `normalizeImageResponse` case is needed. However, the `n` (number of images) parameter is destructured but never forwarded to xAI, so a request for multiple images silently produces one.

The `testUtils.ts` refresh function hardcodes the client ID string instead of importing `SUPERGROK_CONFIG.clientId`, creating a maintenance divergence risk when the client ID changes.

## Hardcoded client ID in testUtils diverges from central config

The `refreshOAuthToken` function in `testUtils.ts` (line 268) hardcodes `"b1a00492-073a-47ea-816f-4c329264a828"` rather than importing `SUPERGROK_CONFIG.clientId`. Every other provider in the same file imports its config from `@/lib/oauth/constants/oauth` (Codex, Claude, Kiro, etc.), but xAI breaks this pattern:

```ts
// testUtils.ts line 268
client_id: "b1a00492-073a-47ea-816f-4c329264a828",
```

If the client ID is rotated during a security incident, `testUtils.ts` will silently use the old value and token refresh will fail only in the connection test path, not in the executor path.

## `n` parameter silently dropped in xAI image body

The `buildImageBody` function destructures `n = 1` from the request body, and other providers forward it (`batch_size: n` for sdwebui, `numImages: n` for nanobanana), but the xAI case constructs:

```ts
return {
  model: model || "grok-imagine-image",
  prompt,
  aspect_ratio: xaiSizeMap[size] || "1:1",
  resolution: "1k",
};
```

If a client requests `n: 4` images from xAI, they get one image back with no indication that the count was ignored. Either pass `n` through (if xAI's API supports it) or return a 400 error for `n > 1`.

## Import route accepts accessToken-only but stores an unrefreshable connection

The import route allows `{ accessToken: "...", expiresIn: 3600 }` without a `refreshToken`. This creates a connection that becomes permanently dead after expiry. `DefaultExecutor.refreshCredentials` returns `null` when `credentials.refreshToken` is falsy, so the connection enters a terminal auth failure state with no recovery path. The route does not warn the caller.

Options:
- Require `refreshToken` (return 400 without it)
- Mark `authState` as `"ephemeral"` so downstream systems know not to attempt refresh

## Upstream error text reflected to client

In the import route, refresh failure returns:

```ts
return NextResponse.json(
  { error: `Token refresh failed: ${errorText}` },
  { status: 401 }
);
```

`errorText` is `await refreshResponse.text()` â€” the raw response body from `auth.x.ai`. OAuth servers sometimes return HTML error pages, JSON with internal error codes, or detailed grant information. Reflecting this verbatim to the API caller is an information leak. Truncate to a reasonable length (e.g., 200 chars) or return a generic message with the HTTP status code only.

## `expiresIn` fallback uses `||` instead of `??`

```ts
let finalExpiresIn = expiresIn || 3600;
// and later:
finalExpiresIn = tokens.expires_in || 3600;
```

The `||` operator treats `0` as falsy. If xAI's token server returns `expires_in: 0` (meaning the token is effectively already expired), the route stores the connection with a 1-hour TTL, masking the real expiry state. Use `?? 3600` (nullish coalescing) to only fall back on `null`/`undefined`.

## Dead format label in IMAGE_PROVIDERS

The xAI entry in `IMAGE_PROVIDERS` declares `format: "xai-image"`, but `buildImageBody` and `normalizeImageResponse` both switch on the `provider` string (`"xai"`), not on `format`. The format string is never referenced anywhere else in the codebase. This is dead config that will confuse a future contributor who tries to add format-based dispatch.
