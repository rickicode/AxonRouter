# Tunnel system extraction into workspace packages with Turbopack NFT elimination

This change extracts the tunnel subsystem into a workspace package (`@axonrouter/tunnel`) with dependency injection, and isolates Node.js filesystem/path/os operations behind a pre-compiled JS facade (`@axonrouter/data-dir`). The motivation is eliminating Turbopack Native File Tracing (NFT) warnings that occurred when Next.js server components imported `node:fs`, `node:path`, and `node:os` at the module level. The approach uses `serverExternalPackages` in `next.config.ts` to exclude these packages from bundling entirely, combined with removing all `/*turbopackIgnore: true*/` comment annotations from the source. The DI pattern (`configureTunnelDeps`/`getDeps`) decouples the tunnel package from app-level services (settings, SQLite, MITM password) so it can live outside the Next.js bundle graph.

**Watch for:** Tunnel API routes have no guarantee that `configureTunnelDeps()` has been called before their handler executes — a cold-start race is possible if Next.js serves a tunnel route before `initializeApp` completes (confirmed). One `turbopackIgnore` comment and `new Function` hacks remain in `open-sse/` (confirmed, pre-existing). Three copies of `getDataDir()` with subtly different behavior exist across the codebase (confirmed).

## High-level view

The `@axonrouter/data-dir` package wraps every Node.js builtin operation (fs, path, os, crypto, child_process) behind named function exports shipped as plain `.js` with a hand-maintained `.d.ts`. Because `serverExternalPackages` tells Next.js to skip bundling these packages, Turbopack never attempts to trace their `node:*` imports. The tunnel package depends on `@axonrouter/data-dir` for all filesystem work and uses a DI registry for app-level concerns (SQLite, settings).

The DI configuration happens at module-evaluation time in `initializeApp.ts`, which runs as a side effect of `initApp.ts` being imported. However, Next.js API route handlers (all tunnel routes) import `@axonrouter/tunnel` directly without importing `initApp.ts`. In production, module evaluation order typically ensures `initializeApp.ts` runs first because the app's layout/page components trigger it, but there is no hard guarantee for API-only cold starts.

The `src/lib/dataDir.ts` module retains its own memoized `getDataDir()` while `packages/data-dir/src/index.js` has its own non-memoized copy, and `src/mitm/paths.ts` has a third CJS copy. These three produce the same path today but could drift silently.

## Tunnel DI timing and API route safety

The `configureTunnelDeps()` call lives at the top level of `src/shared/services/initializeApp.ts`, which executes as a side effect when `src/lib/initApp.ts` is first imported. The tunnel API routes (`src/app/api/tunnel/enable/route.ts`, etc.) import `@axonrouter/tunnel` directly and never import `initApp.ts`. In Next.js App Router, API route modules can be loaded independently — a fresh worker or serverless cold start could theoretically load a tunnel route handler before `initializeApp.ts` has been evaluated.

If `getDeps()` is called before configuration:

```typescript
// packages/tunnel/src/deps.ts
export function getDeps(): TunnelDeps {
  if (!_deps) throw new Error("Tunnel dependencies not configured. Call configureTunnelDeps() first.");
  return _deps;
}
```

This throws synchronously, which the route handler catches and returns as a 500. The system fails closed, but the user gets an opaque error with no recovery path. The fix is either adding `await ensureAppInitialized()` at the top of each tunnel route handler, or using Next.js `instrumentation.ts` (the standard mechanism for server-side initialization that runs before any route).

Confidence: **confirmed** — no import chain connects tunnel route modules to `initApp.ts`.

## getDataDir divergence across three implementations

| Location | Memoized | Uses APPDATA env | Fallback behavior |
|----------|----------|-----------------|-------------------|
| `src/lib/dataDir.ts` | Yes (module-level `_dataDir`) | Yes | Falls back to `AppData/Roaming` |
| `packages/data-dir/src/index.js` | No | Yes | Falls back to `AppData/Roaming` |
| `src/mitm/paths.ts` | No (but `DATA_DIR` const caches) | Yes | Falls back to `AppData/Roaming` |

The `src/lib/dataDir.ts` version is what most of the application uses (imported in `sqliteHelpers.ts`, `localDbStorage.ts`, `sqliteBootstrap.ts`), while the `@axonrouter/data-dir` package's `getDataDir()` is used by the tunnel package via `resolveDataPath()`. If someone changes the app-level `getDataDir()` logic (e.g., to support a `DATA_DIR` env override), the package-level copy won't follow.

Confidence: **confirmed** — three separate `getDataDir` function definitions found via search.

## data-dir as committed pre-compiled JavaScript

`packages/data-dir/src/index.js` is plain ESM JavaScript (not TypeScript) with a manually maintained `index.d.ts`:

- No compiler validates that the `.d.ts` signatures match the `.js` implementation.
- Adding a new helper requires editing both files in lockstep.
- The `readFileAbsolute` function returns `readFileSync(p, enc)` which is `string | Buffer` depending on whether `enc` is provided. In `sqliteBootstrap.ts`, it's used as `readFileAbsolute(DB_JSON_FILE, 'utf-8') as string` — the cast is safe when `enc` is provided, but the type signature permits calling without `enc` and getting a `Buffer` where `string` is expected downstream.

Confidence: **confirmed** — the `.d.ts` and `.js` are both in the tree with no `tsconfig.json` or build script in the package.
